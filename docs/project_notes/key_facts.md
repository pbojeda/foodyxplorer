- **Supabase Pooler тАФ CRITICAL operational rule (discovered 2026-04-16 during release merge #136)**: Supabase transaction pooler (port **6543**) executes `DISCARD ALL` between sessions, resetting `search_path` to empty. This breaks Prisma and Kysely which emit queries without `public.` prefix тЖТ `relation "X" does not exist` errors. **ALL DATABASE_URL env vars MUST use session pooler (port 5432)**, not transaction pooler (port 6543). Fix applied 2026-04-16 on Render API services (`nutrixplorer-api-dev` + `nutrixplorer-api-prod`). `ALTER ROLE postgres SET search_path TO public, extensions` was attempted but pgbouncer overrides it. URL-level `?options=-c search_path=public` also silently ignored by transaction pooler. Only reliable fix: port 5432 (session mode). Trade-off: session pooler has lower concurrent connection limits (~20-50 vs ~200 on transaction mode), sufficient for current traffic. When scaling beyond session pooler limits, options: (a) Prisma `previewFeatures = ["multiSchema"]` (prefixes `public.` in queries), (b) Supabase Team plan (configurable pooler search_path), (c) direct connection with Prisma-side pooling (`connection_limit` in URL). **DIRECT_URL** (for migrations) currently points to `pooler.supabase.com:5432` (session mode via pooler) тАФ functional but ideally should point to `db.<project-ref>.supabase.co:5432` (true direct, no pooler). Documented as tech-debt, not blocking.
