## Spec

### Description

**Origin.** This ticket is a direct spin-off from `F-NLP-CHAIN-ORDERING` (PR #202). During `/review-spec` round 1 on that ticket, Gemini + Codex both independently verified that the original H5-B fix premise was wrong: the existing `detectMenuQuery` (`packages/api/src/conversation/menuDetector.ts:17-24`) only matches **explicit** trigger patterns (`menú:`, `menu:`, `menú del día:`, `de menú:`). It returns `null` for implicit multi-item queries like `"paella y vino"` — which means simply re-running `detectMenuQuery` on F-NLP-stripped text does not solve H5-B. Implementing implicit multi-item detection requires a **new capability**, not a reorder.

User decision 2026-04-23 (documented in `F-NLP-CHAIN-ORDERING` Completion Log): split H5-B off into this ticket rather than expanding `F-NLP-CHAIN-ORDERING` scope mid-flight. Rationale: new detection capability deserves its own full Step 0-6 SDD cycle with dedicated `/review-spec` + `/review-plan`, and the false-positive risk against dish names containing conjunctions is non-trivial (see catalog examples below). This ticket runs as **PR4 of pm-sprint2** (same session as PR1-PR3).

### What to build

A new implicit multi-item detector activated on text that has passed through `extractFoodQuery`'s F-NLP conversational-wrapper strip. Behavioural contract:

**H5-B canonical queries (from `/tmp/qa-dev-2026-04-22.txt` Cat 15, 19) — must return `menu_estimation` with ≥2 items after this ticket:**

- `"he cenado una ración de paella y una copa de vino"` → 2 items (paella + vino)
- `"esta mañana he tomado café con leche y tostada"` → 2 items (café con leche + tostada)
- `"he entrado en un bar y me he pedido una caña y unas bravas"` → 2 items (caña + bravas)

**Catalogue false-positive landmines (must NOT be split):**

- `"arroz con leche"` (single dish — `CE-050`)
- `"pan con tomate"` (single dish — `CE-012`)
- `"mar y montaña"` (single dish — catalan traditional)
- `"huevos y jamón"`, `"costillas con patatas"`, `"macarrones con queso"` etc.
- F-H4 additions (regional dishes) with conjunctions in the name — MUST be audited.

### Scope constraints

1. **Detection runs ONLY on F-NLP-stripped text**, not raw input. The pre-strip first-pass `detectMenuQuery` (F076 contract) remains unchanged — queries with explicit `menú:` triggers continue to route via the existing detector.
2. **No new error codes, no API schema changes.** This is a routing-only change inside `conversationCore.ts`.
3. **Must not regress F076 contract**: existing 48 F076 tests remain green.
4. **Must not regress F-NLP-CHAIN-ORDERING (PR #202)**: all 3723 post-PR-202 tests remain green. New tests take the count higher.
5. **False-positive tolerance is near-zero.** A single broken dish name in the catalog is a worse outcome than missing a few multi-item queries. The detector's default must be "this is a single dish" unless strong positive signal exists.

### API Changes

None. No endpoint signature/schema changes. The `menu_estimation` response shape is already documented.

### Data Model Changes

None.

### UI Changes

None.

### Edge Cases & Error Handling (placeholder — to be expanded in Step 0 Spec round)

- Queries already routed to menu by the first-pass explicit `detectMenuQuery` MUST NOT be re-routed (avoid double-matching).
- Queries containing ONLY a conjunction and a single food name (`"un bocadillo y nada más"`) must stay single-dish.
- Queries with ≥3 items (`"paella, vino y flan"`) must return ≥3 items, not truncate.
- Error-safety parity with PR #202: if the new detector throws, fall back to the F-NLP-CHAIN-ORDERING single-dish path (no 500 to caller).
